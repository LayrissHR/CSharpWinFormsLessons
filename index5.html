Трислоен Модел & Рефакториране

Какво е трислойна архитектура?

Това е архитектурен модел, който разделя приложението на три логически части. Това улеснява поддръжката, тестването и промените. Представете си **Ресторант**:

1. Presentation Layer

"Сервитьорът"

Отговаря за визуализацията и приемането на поръчките от клиента. Не готви храната!

Controllers, Views, HTML, JSON

2. Business Layer

"Готвачът"

Тук се случва магията. Валидации, изчисления, правила. Сърцето на приложението.

Services, Domain Models, Logic

3. Data Layer

"Складът / Килерът"

Знае само как да вземе и запази продукти (данни). Не знае рецепти.

Repositories, SQL, DAO

Какво е Рефакториране?

"Рефакторирането е процес на промяна на софтуерна система по такъв начин, че да **не се променя външното ѝ поведение**, но да се **подобри вътрешната ѝ структура**." — Мартин Фаулър

Инкрементални промени (Baby Steps)

Не се опитвайте да пренапишете цялата програма наведнъж. Това води до счупен код с дни. Вместо това, правете малки, безопасни стъпки.

  • Премести един метод.
  • Пусне тестовете (Зелено!).
  • Преименувай променлива.
  • Пусне тестовете отново.

Цикълът Red-Green-Refactor

Това е ритъмът на безопасното програмиране, особено при TDD (Test Driven Development).

Red
Напиши тест, който се чупи.
Green
Напиши код, за да мине тестът.
Refactor
Почисти кода без да чупиш теста.

Правилото на бойскаута

"Винаги оставяй къмпинг зоната (кода) по-чиста, отколкото си я намерил." Ако видиш малък проблем, оправи го веднага, но само ако имаш тестове!

Проблемът: "Monolithic Function"

Този код нарушава всички правила на трислойния модел. Нека го разгледаме стъпка по стъпка.

01

Проблем: Смесване с UI/HTTP

Функцията зависи от `req` и `res`. Не можете да я извикате от конзолен скрипт или друга част на приложението, без да фалшифицирате HTTP обекти.

02

Проблем: Хардкодната Логика

Валидацията `password < 6` е "зазидана" тук. Ако искате да промените правилото, трябва да редактирате HTTP контролера. Това нарушава Single Responsibility Principle.

03

Проблем: Директен SQL достъп

Функцията създава DB връзка и пише SQL. Това прави тестването невъзможно без реална база данни. Бавно и чупливо.

Bad Practice 
function registerUser(req, res) {
  // 1. UI Dependency
  const username = req.body.username;
  const password = req.body.password;

  // 2. Hardcoded Logic
  if (password.length < 6) {
    return res.error("Too short");
  }

  // 3. Direct DB Access
  const db = new Database();
  const sql = "INSERT INTO users...";
  db.execute(sql);

  return res.success();
}

Анатомия на Рефакторирания Код

Нека разгледаме финалния резултат под лупа. Вместо една голяма функция, вече имаме три специализирани класа. Ето как и защо се промени всичко.

1. Presentation (Controller)

UserController.js 
function register(req, res) {
  // 1. Extract Input
  const user = req.body;

  // 2. Delegate to Business Layer
  userService.validate(user);

  // 3. Delegate to Data Layer
  userRepo.save(user);

  // 4. Return Response
  return res.success();
}

Какво се промени?

  • Вече няма if проверки за пароли.
  • Вече няма SQL заявки.
  • Остана само "оркестрацията" на процеса.

Защо?

Това прави контролера "тънък" (Skinny Controller). Ако сменим правилата за валидация, няма да пипаме този файл. Ако сменим базата данни, няма да пипаме този файл.

2. Business (Service)

UserService.js 
class UserService {
  validate(user) {
    // Pure Logic
    if (!user.username) 
      throw new Error("Missing username");
      
    if (user.password.length < 6) 
      throw new Error("Weak password");
      
    // More complex rules...
  }
}

Какво се промени?

  • Тук живеят всички "Business Rules".
  • Няма req или res (HTTP).
  • Няма SQL.

Защо?

Този код е 100% тестваем (Unit Test). Не ни трябва сървър или база данни, за да проверим дали логиката за паролата работи. Можем да ползваме този клас и в CLI скрипт, не само в Web API.

3. Data (Repository)

UserRepository.js 
class UserRepository {
  save(user) {
    // Database specifics
    const sql = "INSERT INTO users...";
    db.execute(sql, [user.name, user.pass]);
  }
}

Какво се промени?

  • Тук е единственото място със SQL.
  • Изолирано от логиката и UI.

Защо?

Ако утре решим да сменим MySQL с MongoDB, променяме САМО този файл. Останалите 99% от приложението няма да разберат, че нещо се е случило.
